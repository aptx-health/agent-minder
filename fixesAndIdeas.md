## Ideas

These are just thoughts on how we could potentially improve things. They are in no particular order - so don't read into what's first, middle, last. This is just a stream of ideas.\

Active concerns could be better managed. Right now the sonnet agent needs to keep in mind what the current state is for each so it doesn't recreate anything. Not sure if that's efficient/effective. They might have false positives in terms of overlap/redundancy. However, I don't know if that's worth fixing. It might be that our 'solution' makes the problem worse. 
I've looked at the datatypes for the llm_analyzer under analysis.go - It looks like there's a AnalysisConcern type. Maybe when a new one is created, we could automatically calculate a timestamp?I know they're stacked in the UI chronologically, but the agent might find the stamp useful. They could tell which are stale. Right now we've got a mechanical duplicate detector for concerns which makes sense but I'm sure results in occassional false positives.
I'm wondering how "token expensive" it'd be if we simply presented the full list of timestamped concerns to the agent at analysis time - then it recreates the list based on its analysis. That way duplicates are less likely. It'd take more time/tokens for the agent to parse but may end up providing richer, contextual concerns. Looking over the poller.go a bit more, it appears that the analyzer agent (tier 2) returns a list (empty is an option) for concerns that can get appended to the list if there isn't a duplicate detected. It appears that it's getting a list of active concerns. Maybe we can modify the tier 2 analyzer prompt to have it groom the list of concerns? Right now, concerns persist and there's no way for them to get "trimmed/archived" in the event that they are resolved. I recognize giving the agent complete control over the list would limit our options for programttic featuers like timestamps but that's ok.

Checking out poller.go - system prompts for tier 2 analysis. It looks like we're not giving the agent options for concern levels - it only sees "warning" in the example JSON structure. Perhaps we should provide options: INFO, WARN, DANGER which would be color coded in the TUI list.

From a agent-msg perspective - it might be nice to quickly onboard a new agent with a generated guidance message from the agent-minder app? Like there could be a living "onboard" topic that's essentially a summary of the goal of the work, agents already involved and their focus, summary of recent work, etc.. It'd be expensive in terms of time and tokens to run at every poll though. So it could be something the user queues up. It could be very similar to how (u) user message works. Press (o) onboard and then the user has the OPTION to include multi-line input to help guide the sonnet agent with crafting the onboard message that gets thrown into the onboarding topic. For instance, the user will know "I need a new agent who will focus on writing tests for feature A", so they can inform the onboard message writing agent in the minder app to craft the onboarding message specifically for this type of agent. If the user doesn't provide guidance, then the minder app would generate a generic message that just gets the new agent(s) up to speed on the project and current progress. I guess the "onboarding" topic would only have one, most-recent message. I realize most of this work would be in the agent-minder code. From a user's perpective, we'd create a new agent in the desired repo (maybe a worktree) and tell it to check the "project/onboarding" topic to get started. I know that we have an automitic detection of new worktrees in the minder app, but I haven't tested that feature yet.
Anyway, telling the agent where to check is a manual step. There's no way for the agent to check the list of active projects under the agent-minder, unless of course we build a Claude skill.. However, I don't know if a skill is warranted at this point. I need to try using the onboarding feature before going down the skill route.
